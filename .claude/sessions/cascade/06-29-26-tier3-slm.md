# Session: Tier 3 — SLM Classification (CLOSED)

## Problem

25,481 actors flagged as `pending_slm` by reconciliation need SLM classification. On local CPU at ~0.3/s this is ~24 hours. On RunPod GPU at ~3-6/s this is ~1-2 hours at ~$1.40.

## RunPod SLM batch plan

Self-contained Python script on RunPod that:

1. Connects to Postgres via SSH tunnel (local Postgres exposed to pod)
2. Installs Ollama + loads GGUF model on the pod
3. Queries `pending_slm` actors with provision text from Postgres
4. Calls Ollama /api/chat for each actor
5. Writes `slm_drrp` / `slm_position` back to provision_actors via Postgres

No Rust build needed on pod. No taxa slm command needed. Just Python + Ollama + Postgres.

### Setup

1. Create RTX 4090 pod ($0.69/hr), PyTorch template
2. SSH tunnel: `ssh -L 5433:localhost:5433 -p <PORT> root@<IP>` (exposes local Postgres to pod)
3. Install Ollama on pod: `curl -fsSL https://ollama.com/install.sh | sh`
4. Transfer GGUF + Modelfile, create model: `ollama create gemma3-position -f Modelfile`
5. Upload + run batch script

### Alternative: simpler

Instead of SSH tunnel, just:
1. `runpodctl send` the GGUF to the pod
2. Upload the batch script with Postgres connection details hardcoded to localhost (via tunnel)
3. Open SSH tunnel in a separate terminal
4. Run the script

## Experiment: SLM on all actors — could it replace the classifier?

**Hypothesis**: if SLM performs well on all actors (not just pending_slm hard cases), the pipeline could simplify from 4 tiers to 2:

```
Current:  Regex → Dep features → Classifier → Reconcile → SLM → Reconcile
Proposed: Regex → SLM (all actors)
```

Eliminates: dep features (spaCy batch), embeddings (for classification), classifier, reconciliation logic.

**Test**: run SLM on a sample of `reconciled_agree` actors (the easy cases where regex+classifier already agree at 79%). If SLM matches 90%+, classifier adds no value.

**Cost/time**: RTX 5090 at 12.5 actors/s, 4 workers. Full 69K corpus would take ~90 minutes, ~$1.50. Cheaper than maintaining the classifier pipeline.

**Why this matters**: dep features are a Python/spaCy dependency that doesn't integrate into the Rust pipeline. Embeddings take hours on CPU. Removing both would massively simplify the customer batch parse flow.

**Constraint**: this experiment is about simplifying the **hub enrichment pipeline** only. The edge architecture (LanceDB, ONNX embeddings, local-first inference on devices) is unchanged. Embeddings are still needed for edge vector search even if the hub classifier is replaced by SLM. LanceDB remains the edge store.

**Decision**: run after Tier 3 completes. Compare SLM accuracy vs reconciled accuracy on agree+cls_confident actors.
